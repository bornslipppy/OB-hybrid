## 2. DIY Approach: PM + Cursor

### 2.1 Overview: Product Managers Using Cursor Directly

**What This Approach Looks Like:**
Product managers are increasingly using Cursor (AI-first code editor) to write PRDs, generate specifications, and even prototype features directly in markdown files treated as source code.

**Real-World Adoption:**
- Senior PMs at companies like Chime use Cursor to automate PRD creation, Jira ticket generation, and rapid prototyping
- Markdown PRDs become "single source of truth" with version control (git)
- AI agents (Cursor's LLMs) generate production-ready specs with context from project files

**Sources:**

### 2.2 How PMs Use Cursor: Recommended Workflow

**6-Phase PRD Workflow:**

1. **Choose Template:** Start with structured PRD markdown template
2. **Build Context:** Create context files (company info, user research, market data)
3. **Sharpen Thinking:** Use Socratic/strategic AI prompting to refine ideas
4. **Generate Drafts:** AI generates 2–3 PRD draft options
5. **Multi-Perspective Review:** Run reviews with AI personas (engineer, designer, executive)
6. **Iterate:** Address feedback and refine specifications

**Technical Implementation:**
- Use `@-mentions` to reference context files (research, company docs, codebase)
- Reviewer persona files provide targeted feedback
- Model Context Protocols (MCPs) enable:
  - Publishing markdown PRDs to Confluence/Notion
  - Auto-generating Jira epics/stories with acceptance criteria
  - Drafting status reports
  - End-to-end PM lifecycle inside Cursor

**PRD Structure for AI-Readability:**
- Clear section headings (Problem, Solution, User Stories, Technical Reqs, Acceptance Criteria, Constraints)
- Concise bulletized user stories
- Explicit, testable acceptance criteria
- Isolated requirements for easier parsing

### 2.4 Critical Limitations & Challenges

**For Non-Technical PMs (The Target Audience):**

**1. Developer-Centric UX**
- **Challenge:** Cursor is designed for engineers, not PMs
- **Impact:** Steep learning curve for non-technical users
- **Specific Friction:** IDE interface, command-line concepts, git workflows
- **Evidence:** PMs report difficulty with repo-connected tools without engineering background

**2. No Product Discovery Framework**
- **Challenge:** Cursor is execution-focused (coding), not discovery-focused
- **Impact:** No built-in frameworks for understanding "what to build"
- **Gap:** PMs must manually structure discovery work outside Cursor
- **Missing:** Customer interview analysis, behavioral data triangulation, say-vs-do detection

**3. Lack of Evidence Traceability**
- **Challenge:** AI-generated specs are disconnected from customer evidence
- **Impact:** No automatic linking of decisions to customer quotes, usage metrics, or business goals
- **Consequence:** Trust issues with engineering teams (where did this requirement come from?)
- **Wozy Differentiator:** Built-in BMAD reasoning with evidence chains

**4. Prototype-to-Product Gap**
- **Challenge:** AI prototypes often standalone, not connected to real codebase
- **Impact:** Handoff friction between PM prototype and engineering implementation
- **Workaround Needed:** Repo-aware tools or tight designer/engineer pairing
- **Source:** [Builder.io: AI Prototyping for PMs](https://www.builder.io/blog/ai-prototyping-product-managers)

**5. Complexity Management**
- **Challenge:** No visual, discovery-oriented interface for managing product complexity
- **Impact:** Difficult to visualize relationships between features, customers, and business goals
- **Linear Workflow:** Coding mindset (sequential) vs. PM mindset (systems thinking, exploration)

**6. Context Persistence Issues**
- **Challenge:** Unlike purpose-built PM tools, Cursor lacks PM-specific context management
- **Impact:** PMs must manually maintain context across sessions
- **Comparison:** Traditional PM tools keep customer feedback, roadmaps, and decisions in one place

**7. Trust & Governance Risks**
- **Challenge:** AI hallucinations and quality control concerns
- **Impact:** Need for peer review, CI/tests, explicit governance policies
- **Enterprise Friction:** ~85% of companies plan to customize autonomous agents but lack mature policies
- **Source:** [Orchestrator's Era: 2026 State of AI Agents](https://redreamality.com/blog/ai-agents-in-product-management-2026/)

**8. Speed Mismatch (PM Bottleneck)**
- **Challenge:** AI-augmented engineering outpaces PM validation and research cycles
- **Impact:** PMs become bottleneck as engineers move faster with AI
- **Consequence:** Pressure to rush discovery work, potentially leading to poor product decisions
- **Source:** [Ditto: Cursor for PMs](https://askditto.io/news/ditto-is-cursor-for-product-managers)

---

### 2.5 Practical Mitigations (2026 Best Practices)

**What Advanced PMs Do:**
1. **Structured Workflows:** Follow disciplined processes (Create Issue → Explore → Plan → Execute → Review → Peer Review → Update docs)
2. **Enforce Quality Gates:** CI/tests, peer review, explicit agent governance
3. **Use Project Rules:** `.cursorrules` files to keep outputs consistent and auditable
4. **Pair with Repo-Aware Tools:** Combine Cursor with visual prototyping tools that capture live product state

**Sources:**
- [Medium: Non-Technical People Shipping Code with AI](https://medium.com/@sangam.n.pandey/how-non-technical-people-are-shipping-production-code-using-ai-fb7b906a6339)

### 3.2 Product Manager Pain Points (Writing Specifications)

**Research Source:** Analysis of PM best practices, pain point articles, and industry guides from 2026.

---

**Pain Point #1: Ambiguity & Vague Scope**

**Issue:** Specs lack clarity, causing rework and missed expectations

**Root Cause:**
- PMs struggle to articulate problem and business impact before jumping to features
- Missing "why" and success metrics
- **Consequence:** Engineering builds wrong thing, requiring late-stage redesign

**Evidence from Best Practices:**
> "Ambiguous specs → rework and missed expectations. Fix: state problem and business impact before features; use concise context and success metrics."

**How Wozy Addresses This:**
- BMAD reasoning forces explicit problem definition
- Triangulation of customer SAY, DO, technical feasibility, PM beliefs
- Every decision traced to evidence

**Source:** [River Editor: How to Write PRDs Engineering Teams Love](https://rivereditor.com/guides/how-to-write-product-requirements-documents-2026)

---

**Pain Point #2: Missing Testable Acceptance Criteria**

**Issue:** Endless "is it done?" debates between PM and engineering

**Root Cause:**
- Specs lack measurable, testable acceptance criteria
- Vague definitions of "done"
- **Consequence:** Scope creep, prolonged QA cycles, finger-pointing

**Evidence:**
> "Missing testable acceptance criteria → endless 'is it done?' debates. Fix: include measurable, testable acceptance criteria (SLA numbers, limits)."

**How Wozy Addresses This:**
- Machine-executable specs require explicit acceptance criteria
- Cursor-ready format forces precision
- Definition-of-done embedded in output

**Source:** [Parallel HQ: How to Write Product Requirements 2026](https://www.parallelhq.com/blog/how-to-write-product-requirements)

---

**Pain Point #3: Over-Specifying Implementation**

**Issue:** PMs micromanage engineers, blocking better technical solutions

**Root Cause:**
- Specs dictate "how" instead of "what" and "why"
- Engineering feels constrained
- **Consequence:** Suboptimal technical solutions, demoralized engineers

**Evidence:**
> "Over‑specifying implementation → blocks better technical solutions. Fix: answer 'what' and 'why,' leave 'how' to engineering."

**How Wozy Addresses This:**
- Focus on outcomes and business logic
- Codebase-aware but not prescriptive
- Leaves technical implementation decisions to engineering

**Source:** [Atlassian: What is a PRD?](https://atlassian.com/agile/product-management/requirements)

---

**Pain Point #4: Edge Cases & Error States Omitted**

**Issue:** Production bugs and late fixes from undocumented edge cases

**Root Cause:**
- PMs focus on happy path, neglect failure scenarios
- **Consequence:** Bugs discovered in production, emergency patches

**Evidence:**
> "Edge cases and error states omitted → production bugs and late fixes. Fix: document edge cases (file formats, scale, failure modes)."

**Source:** [River Editor PRD Guide](https://rivereditor.com/guides/how-to-write-product-requirements-documents-2026)

---

**Pain Point #5: Weak Stakeholder Alignment**

**Issue:** Siloed discovery leads to misaligned assumptions and wasted cycles

**Root Cause:**
- PM works in isolation during discovery
- Design and engineering brought in too late
- **Consequence:** "This won't work technically" or "This UX doesn't scale"

**Evidence:**
> "Weak stakeholder alignment / siloed discovery → misaligned assumptions and wasted cycles. Fix: involve design/engineering in research and interviews; make PRD collaborative and living."

**Source:** [Ulad Shauchenka: How to Write a Good PRD](https://www.uladshauchenka.com/p/how-to-write-a-good-product-requirements)

---

**Pain Point #6: Engineering Distrust / Lack of Traceability**

**Issue:** Engineers don't trust PM's reasoning, causing friction and rework

**Root Cause:**
- Specs lack evidence chains (where did this requirement come from?)
- Assumptions not explicit or testable
- **Consequence:** "Show me the data" debates, delayed implementation

**Evidence:**
> "Engineers distrust / lack of traceability → friction and rework. Fix: make assumptions explicit, convert them to testable hypotheses and trace decisions to data; use KPI dictionaries and decision rules."

**How Wozy Addresses This (Core Differentiator):**
- Every decision linked to customer quote + usage metric
- Traceable evidence chains built-in
- Say-vs-do contradiction detection
- BMAD reasoning = explicit, auditable decision-making

**Source:** [Routine: AI Prompts to Write PRDs Engineers Trust](https://routine.co/blog/posts/ai-prompts-prd-engineers)

---

**Pain Point #7: Non-Functional & Compliance Requirements Missed**

**Issue:** Late rewrites in regulated/enterprise contexts

**Root Cause:**
- Specs focus on features, ignore performance, security, RBAC, regulatory constraints
- **Consequence:** Late-stage architecture changes, compliance failures

**Evidence:**
> "Non‑functional, compliance, and RBAC requirements missed → late rewrites in regulated/enterprise contexts. Fix: capture performance, security, roles, and regulatory constraints up front."

**Source:** [Parallel HQ: Product Requirements Guide](https://www.parallelhq.com/blog/how-to-write-product-requirements)

## 5. Wozy Differentiation Strategy

### 5.1 Core Positioning

**Tagline:** "The Product Reasoning Infrastructure for the AI-Native Era"

**One-Sentence Pitch:**

### 5.2 Three-Pillar Differentiation

#### Pillar 1: Infrastructure, Not Tool

**Traditional PM Tools:** Feature sets for human consumption (roadmaps, dashboards, feedback boards)

**Wozy:** Foundational primitives for machine-client world
- `ingest` - Add signals to knowledge graph
- `reason` - Run BMAD analysis
- `generate` - Create machine-executable specs
- `trace` - Follow evidence chains

**Why This Matters:**
- As AI agents take over execution, structured reasoning becomes mandatory
- Wozy isn't a tool bolted onto existing workflows—it's the operating system for product development

---

#### Pillar 2: Evidence-Based Reasoning (BMAD Method)

**Traditional PM Tools:** PMs manually connect dots between feedback, data, and decisions

**Wozy:** Automatic triangulation of 4 signals
1. What customers **SAY** (Modjo transcripts, Zendesk tickets)
2. What customers **DO** (Datadog, Looker behavioral data)
3. What's **TECHNICALLY** possible (GitHub codebase context)
4. What PM **BELIEVES** matters (strategic brief)

**Unique Capability:** Say-vs-do contradiction detection
- Example: Customers say they want feature X, but usage data shows they don't use similar features
- Wozy flags this contradiction, preventing wasted development

**Why This Matters:**
- Engineering teams trust specs with evidence chains
- Reduces "show me the data" debates
- Prevents building features customers claim to want but won't use

---

#### Pillar 3: Machine-Executable Specs (Paradigm Bridge)

**Traditional PM Tools:** Human-readable PRDs (Notion, Confluence, Word docs)

**Cursor DIY:** Machine-readable specs (but requires developer skills)

**Wozy:** Machine-executable specs for non-technical PMs
- Structured (YAML + markdown Cursor can parse)
- Specific (file paths, component names, data model changes)
- Contextual (WHY decisions made, traced to evidence)
- Codebase-aware (references actual files, patterns)
- Definition-of-done explicit (acceptance criteria, verification)

**Why This Matters:**
- Cursor/Claude Code can execute without reformatting
- 10x faster handoff to engineering
- PM reasoning becomes the single source of truth
